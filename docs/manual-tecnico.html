1. Stack Tecnológica e Visão Geral

Tecnologias utilizadas e arquitetura do sistema

Next.js 15

App Router, Server Components, API Routes. React 19. TypeScript estrito.

PostgreSQL + Prisma

ORM com type-safety. Migrations versionadas. Schema declarativo.

NextAuth.js v5

Credentials provider. JWT com claims customizados (role, isTrainee).

Tailwind CSS 4

Utilitários CSS. Tema configurado em CSS vars. Build otimizado.

Bcrypt

Hash de senhas com salt. Nenhuma senha armazenada em texto plano.

Yarn

Gerenciador de pacotes. Lockfile versionado no repositório.

Fluxo de Request

Browser
→
Middleware (auth + trainee)
→
Next.js Server
→
Prisma ORM
→
PostgreSQL

Padrão de renderização

  • Server Components: todas as páginas (pages) buscam dados diretamente via Prisma — sem fetch client-side para dados iniciais.
  • Client Components: grids interativos (mesas, produtos, estoque) marcados com "use client".
  • API Routes: endpoints em src/app/api/ para todas as mutações.
  • force-dynamic: páginas com dados em tempo real usam export const dynamic = "force-dynamic" para evitar cache.

6. Autenticação e Controle de Acesso

NextAuth, JWT, middleware e proteção de rotas

Fluxo de autenticação

POST /api/auth/signin
→
Credentials Provider
→
bcrypt.compare()
→
JWT callback
→
Session cookie

Callbacks em src/auth.ts

src/auth.ts — JWT callback
jwt: async ({ token, user }) => {
  if (user) {
    token.role      = (user as any).role;
    token.isTrainee = (user as any).email === "treinamento";
  }
  return token;
},
session: async ({ session, token }) => {
  (session.user as any).role      = token.role;
  (session.user as any).isTrainee = token.isTrainee;
  return session;
}

Middleware de proteção (src/middleware.ts)

O middleware é executado em todas as requisições antes do handler. Ele realiza 3 funções:

1. Redirecionar não autenticados

Qualquer rota não pública sem sessão válida → redirect para /login.

2. Restringir Caixa

Role CAIXA só acessa /, /mesas, /produtos, /estoque. Demais → redirect.

3. Interceptar Trainee

Se isTrainee=true e método ≠ GET → retorna { ok: true, _treinamento: true } sem chegar ao banco.

src/middleware.ts — intercepção trainee
const isTrainee = token?.isTrainee === true;
const isApi     = req.nextUrl.pathname.startsWith("/api");
const method    = req.method;

if (isTrainee && isApi && method !== "GET") {
  return NextResponse.json({ ok: true, _treinamento: true });
}

Como trocar a senha de um usuário

Não há tela de troca de senha. Use o Prisma Studio ou um script Node:

script: reset-senha.ts
import { PrismaClient } from "@prisma/client";
import bcrypt from "bcryptjs";

const prisma = new PrismaClient();
const hash = await bcrypt.hash("NOVA_SENHA", 12);
await prisma.user.update({
  where: { email: "admin" },
  data:  { senhaHash: hash },
});
await prisma.$disconnect();

8. Modo Treinamento — Implementação

Como a simulação em memória foi construída

Camadas da implementação

Camada 1 — Middleware

Intercepta qualquer mutação de API (!= GET) e retorna { ok: true } sem tocar no banco.

Camada 2 — Estado local

simulado: Record<mesaId, MesaComPedido> em useState no componente mesas-grid.

Camada 3 — Dados efetivos

mesaEfetiva mescla estado real do servidor com o simulado do trainee.

Camada 4 — UI

Cards e modal usam display e mesaEfetiva para mostrar dados corretos sem consultar o servidor.

Estrutura do estado simulado

src/app/mesas/mesas-grid.tsx
// Estado de simulação em memória (só existe para trainee)
const [simulado, setSimulado] = useState<Record<string, MesaComPedido>>({});

// Dados efetivos que o UI usa
const mesaEfetiva =
  (isTrainee && mesaIdSelecionada && simulado[mesaIdSelecionada])
    ? simulado[mesaIdSelecionada]
    : mesaSelecionada;

// Exemplo: abrirMesa no modo trainee
if (isTrainee) {
  atualizarSimulado(mesa.id, {
    ...mesa,
    status: "OCUPADA",
    orders: [{
      id: `treino-${Date.now()}`,
      total: 0,
      paymentStatus: "PENDENTE",
      items: [],
    }],
  });
  return;  // não chama a API
}
ℹ️
Limitação por design O estado simulado é volátil (in-memory). Recarregar a página zera a simulação — o que é intencional para o modo treinamento.

9. Lógica de Pagamento Split

Como o pagamento dividido é processado e armazenado

Helper resolverPagamentos

Presente em ambos os endpoints de fechar (fechar/route.ts e fechar-e-liberar/route.ts):

type PagamentoEntry = { forma: string; valor: number };

function resolverPagamentos(pagamentos: PagamentoEntry[], totalFinal: number) {
  const validos = pagamentos.filter(p => p.valor > 0);

  // Pedido gratuito (desconto 100%)
  if (validos.length === 0)
    return { formaPagamento: pagamentos[0]?.forma ?? "DINHEIRO", pagamentosSplit: null };

  // Forma primária = maior valor pago
  const primario = validos.reduce((a, b) => b.valor > a.valor ? b : a);

  // Salva split só se houver mais de uma forma
  const pagamentosSplit = validos.length > 1 ? validos : null;

  return { formaPagamento: primario.forma, pagamentosSplit };
}

Leitura no Financeiro

A página src/app/financeiro/page.tsx distribui os valores corretamente:

type PagEntry = { forma: string; valor: number };

const todasEntradas: PagEntry[] = pedidosFechados.flatMap(p => {
  const split = p.pagamentosSplit as PagEntry[] | null;
  if (split && split.length > 0) return split;          // usa split
  return [{ forma: p.formaPagamento ?? "DINHEIRO", valor: Number(p.total) }];
});

const porForma = {

11. Gestão de Usuários

Como criar, editar e remover usuários no sistema

⚠️
Não há tela de admin de usuários O gerenciamento de usuários é feito via Prisma Studio ou scripts Node.js diretamente no banco. Qualquer alteração requer acesso ao servidor.

Criar novo usuário

import { PrismaClient } from "@prisma/client";
import bcrypt from "bcryptjs";

const prisma = new PrismaClient();

await prisma.user.create({
  data: {
    nome:      "Nome do Funcionário",
    email:     "nome_login",          // sem espaços, sem @
    senhaHash: await bcrypt.hash("senha123", 12),
    role:      "CAIXA",              // ou "ADMIN"
  },
});

await prisma.$disconnect();

Listar usuários existentes

# Via Prisma Studio (interface visual)
npx prisma studio
# Abra a tabela "User" no navegador em localhost:5555

# Via SQL direto
SELECT id, nome, email, role FROM "User";

Regras do campo email (username)

  • O campo email é usado como username de login (não precisa ser um e-mail real).
  • Deve ser único no banco.
  • O valor exato "treinamento" ativa automaticamente o modo treinamento via auth.ts.
  • Não use espaços ou caracteres especiais — afeta o matching do middleware.

Alterar role de um usuário

await prisma.user.update({
  where: { email: "nome_login" },
  data:  { role: "ADMIN" },  // ou "CAIXA"
});

12. Resolução de Problemas (Troubleshooting)

Erros conhecidos, causas e soluções

Erros de banco de dados

SintomaCausa ProvávelSolução
Página em branco / erro 500 DATABASE_URL incorreta ou banco offline Verificar .env e conexão com o PostgreSQL. Testar com npx prisma db pull.
P2021: Table does not exist Migrations não aplicadas Rodar npx prisma migrate deploy
P2002: Unique constraint failed Tentativa de criar item com email/número já existente Verificar dados duplicados na tabela em questão
EPERM ao rodar prisma generate Processo node travou o arquivo DLL do query engine Stop-Process -Name node -Force (PowerShell), depois prisma generate

Erros de autenticação

SintomaCausa ProvávelSolução
Login inválido para usuário correto Senha ou email incorreto no banco Resetar senha via script bcrypt. Verificar campo email sem espaços.
isTrainee sempre false Campo email do usuário treinamento não é exatamente "treinamento" Verificar no banco: SELECT email FROM "User" WHERE nome LIKE '%rein%'
Sessão expira muito rápido AUTH_SECRET mudou ou expiração padrão do JWT Verificar AUTH_SECRET estável. Ajustar session.maxAge em auth.ts se necessário.
Redirect loop no login NEXTAUTH_URL diferente da URL real do sistema Corrigir NEXTAUTH_URL no .env para URL exata de acesso.

Erros de aplicação / UI

SintomaCausa ProvávelSolução
Botão "Fechar Conta" desabilitado Saldo restante ≠ 0 (pagamentos não cobrem o total) Verificar se a soma dos pagamentos iguala o total com desconto (tolerância ±R$0,01).
Mesa aparece ocupada sem pedido Status dessincronizado (falha durante liberação) Usar "Liberar Mesa de Emergência" na UI ou: UPDATE "Table" SET status='LIVRE' WHERE numero=X;
Trainee: mesa não abre isTrainee não está chegando à sessão Verificar callback JWT em auth.ts e email do usuário no banco.
Produtos não aparecem no modal de mesas Nenhum produto cadastrado ou erro ao carregar Verificar tabela Product no banco. Checar logs do servidor.
Financeiro com valores zerados Nenhum pedido fechado no período selecionado Verificar filtro de datas. Confirmar se há pedidos com paymentStatus=PAGO.

Comandos úteis de diagnóstico

# Ver pedidos abertos
SELECT o.id, t.numero, o."paymentStatus", o.total
FROM "Order" o
JOIN "Table" t ON t.id = o."mesaId"
WHERE o."paymentStatus" = 'PENDENTE';

# Verificar status de todas as mesas
SELECT numero, status FROM "Table" ORDER BY numero;

# Forçar liberação de mesa específica (emergência)

# Ver últimos 10 pedidos fechados
SELECT id, total, "formaPagamento", "closedAt"
FROM "Order"

# Ver usuários cadastrados
SELECT nome, email, role FROM "User";
